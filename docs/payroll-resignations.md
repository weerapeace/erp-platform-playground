# Payroll Resignations

หน้าแจ้งลาออกใช้สำหรับเก็บคำขอลาออกของพนักงานก่อนเปลี่ยนข้อมูลจริงในระบบ

หลักการสำคัญ:

- สร้างคำขอไว้ก่อน โดยเก็บใน `employee_portal_requests`
- ตารางนี้ปัจจุบันอนุญาต `request_type` แค่บางค่า จึงเก็บลาออกเป็น `request_type = profile_update` และแยกชนิดจริงด้วย `payload.request_kind = resignation`
- ข้อมูลเฉพาะของการลาออก เช่น วันที่แจ้ง, วันทำงานวันสุดท้าย, เหตุผล, หมายเหตุส่งมอบงาน เก็บใน `payload`
- ตอนสร้างคำขอ ระบบยังไม่เปลี่ยนสถานะพนักงาน
- เมื่ออนุมัติเท่านั้น ระบบจะตั้ง `employees.employment_status = resigned`, ใส่ `employees.resign_date`, และปิดสัญญาปัจจุบันของพนักงาน
- ปฏิเสธหรือยกเลิกจะเปลี่ยนสถานะคำขอเท่านั้น ไม่แตะข้อมูลพนักงานหรือสัญญา
- ทุก action สำคัญเขียน audit ผ่าน `writeAudit`

ของกลางที่ใช้:

- `DataTable` สำหรับตารางรายการ
- `ERPModal` สำหรับฟอร์มและ modal ดำเนินการ
- `ConfirmDialog` สำหรับยืนยันสร้างคำขอ
- `guardPayroll` สำหรับตรวจสิทธิ์ API
- `writeAudit` สำหรับประวัติการทำรายการ

Flow หน้าจอ:

1. กดเพิ่มคำขอแจ้งลาออก
2. เลือกพนักงานจาก dropdown ที่ค้นหาได้
3. บันทึกเป็นคำขอรอตรวจ
4. คลิกแถวในตารางเพื่อเปิดรายละเอียดคำขอ
5. ถ้าคำขอยังรอตรวจ ให้กดอนุมัติ, ปฏิเสธ, หรือยกเลิกจาก modal รายละเอียดได้
6. ก่อนยืนยัน ระบบจะแสดงผลกระทบให้เห็น เช่น อนุมัติแล้วจะเปลี่ยนสถานะพนักงานและปิดสัญญาปัจจุบัน

ข้อจำกัดรอบแรก:

- ยังไม่มี transaction กลางระดับ database สำหรับการอนุมัติหลายตารางพร้อมกัน เพราะรอบนี้ไม่แก้ schema/database function
- ถ้าต้องการความแน่นระดับ production มากขึ้น ควรเพิ่ม RPC/migration แบบ non-destructive เพื่อทำอนุมัติลาออกใน transaction เดียว
